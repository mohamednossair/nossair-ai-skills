# 6. Data & Reporting

**Project**: nossair-ai-skills | **Module**: nossair-ai-skills | **Version**: 1.0

## Data Entities

### Skill Definitions

**Description**: Markdown files that define coding standards, review checklists, and best practices for specific technologies.

**Key Fields**:
- Technology name (e.g., java, typescript, python)
- Coding standards and patterns
- Review checklist items
- Best practices
- Anti-patterns to avoid

**Storage Location**:
- `.junie/skills/<tech>/SKILL.md` (detailed skill definitions)
- `.junie/skills/<tech>.md` (summary references)
- `.claude/skills/<tech>-review.md` (Claude-specific review commands)
- `.windsurf/memories/<tech>-standards.md` (Windsurf-specific standards)

**Business Purpose**: Provide AI assistants with technology-specific guidance to ensure code quality and consistency.

---

### Command Workflows

**Description**: Markdown files that define the step-by-step instructions for unified commands.

**Key Fields**:
- Command name (e.g., review, plan, test-generate, git-commit)
- Description (frontmatter)
- Step-by-step instructions
- Input parameters (e.g., file path, task description)
- Output format requirements

**Storage Location**:
- `.junie/commands/<command>.md` (Junie-specific)
- `.windsurf/workflows/<command>.md` (Windsurf-specific)
- `.claude/skills/<command>.md` (Claude-specific)

**Business Purpose**: Ensure consistent behavior of unified commands across different AI assistants.

---

### Coding Guidelines

**Description**: Project-wide coding standards that apply across all technologies.

**Key Fields**:
- Language-specific standards (Java, TypeScript, Python)
- Framework-specific patterns (Spring Boot, Angular)
- General engineering principles (SOLID, DRY, Clean Code)
- Git workflow rules
- Testing standards

**Storage Location**:
- `.junie/guidelines.md` (comprehensive guidelines)
- `.windsurf/memories/coding-standards.md` (coding standards)
- `.windsurf/memories/tech-stack.md` (technology stack)
- `.windsurf/memories/testing-standards.md` (testing patterns)
- `.windsurf/memories/spring-angular-patterns.md` (framework patterns)

**Business Purpose**: Provide a single source of truth for coding standards across the organization.

---

### Speckit Artifacts

**Description**: Files generated by the Speckit toolkit for specification-driven development.

**Key Fields**:
- Specification (goals, acceptance criteria, data model, open questions)
- Implementation plan (milestones, deliverables, dependencies, risks)
- Task list (dependency-ordered tasks with acceptance criteria)
- Validation report (consistency checks, traceability)
- Execution runbook (branch strategy, commit slices, test scaffolds, review checkpoints)

**Storage Location**:
- `feature/spec/<id>-<slug>/spec.md` (feature specification)
- `feature/spec/<id>-<slug>/plan.md` (implementation plan)
- `feature/spec/<id>-<slug>/tasks.md` (task list)
- Generated in user's project directory (not in this repository)

**Business Purpose**: Provide a structured, traceable approach to feature development from specification to implementation.

---

### Deployment Scripts

**Description**: Shell and PowerShell scripts that automate skill deployment to different AI assistants.

**Key Fields**:
- Source directories
- Target directories
- File/folder copying logic
- Error handling
- User feedback messages

**Storage Location**:
- `scripts/install.sh` (Linux/macOS unified install)
- `scripts/install.ps1` (Windows unified install)
- `scripts/sync-claude.sh` (Linux/macOS Claude sync)
- `scripts/sync-claude.ps1` (Windows Claude sync)
- `scripts/sync-codex.sh` (Linux/macOS Codex sync)
- `scripts/sync-codex.ps1` (Windows Codex sync)

- `scripts/new-skill.ps1` (Windows skill scaffolding)
- `scripts/check-command-parity.sh` (Linux/macOS parity validation)
- `scripts/check-command-parity.ps1` (Windows parity validation)

**Business Purpose**: Automate the distribution of skills to AI assistants, reducing manual effort and errors.

## Data Flow

### Skill Authoring Flow

1. Engineering lead identifies need for new technology support or updated standards
2. Engineering lead edits or creates skill definition files in `.junie/skills/`
3. Engineering lead updates corresponding Claude and Windsurf files
4. Engineering lead updates `.junie/guidelines.md` if needed
5. Engineering lead runs sync scripts to deploy changes to Claude Code and Codex
6. Changes are immediately available to developers using those AI assistants

### Command Update Flow

1. Maintainer identifies need to update a unified command
2. Maintainer updates command definition in `.claude/skills/`
3. Maintainer updates corresponding command in `.junie/commands/`
4. Maintainer updates corresponding workflow in `.windsurf/workflows/`
5. Maintainer updates `.windsurf/memories/commands-reference.md`
6. Maintainer runs parity validation script
7. Maintainer runs sync scripts to deploy changes
8. Changes are immediately available across all AI assistants

### Speckit Workflow Flow

1. Developer provides feature description
2. Developer invokes `/speckit.specify` to generate specification
3. Specification is saved to `feature/spec/<id>-<slug>/spec.md`
4. Developer invokes `/speckit.plan` to generate implementation plan
5. Plan is saved to `feature/spec/<id>-<slug>/plan.md`
6. Developer invokes `/speckit.tasks` to generate task list
7. Tasks are saved to `feature/spec/<id>-<slug>/tasks.md`
8. Developer invokes `/speckit.validate` to check consistency
9. Developer invokes `/speckit.implement` to generate execution runbook
10. Developer follows runbook to implement feature
11. Developer uses unified commands (/review, /test-generate, /git-commit) during implementation

## What Users See vs. What Is Stored

### Developer Experience

**What Developer Sees**:
- AI assistant responses with code reviews, plans, tests, or commit messages
- Consistent command behavior across different AI assistants
- Coding standards applied automatically by AI assistants
- Structured workflows for feature development

**What Is Stored**:
- Markdown files with skill definitions and workflows
- Shell/PowerShell scripts for deployment
- No runtime database or application server
- No user accounts or authentication
- No analytics or usage tracking

### Engineering Lead Experience

**What Engineering Lead Sees**:
- Repository with markdown files and scripts
- Ability to edit skills and commands
- Scripts to deploy changes to AI assistants
- Validation scripts to ensure consistency

**What Is Stored**:
- Source markdown files in version control
- Deployment scripts
- No configuration management system
- No deployment pipeline (beyond CI for parity validation)

## Data Retention

### Current Policy

**No Data Retention Policy**: The system is a static repository of markdown files and scripts. There is no user data, no runtime data, and no analytics data to retain.

**Version Control**: All files are stored in Git with full history. Historical versions of skills and commands are preserved indefinitely.

## Data Security

### Current Security Measures

**No Sensitive Data**: The repository contains only coding standards, workflows, and scripts. No passwords, API keys, or personal information.

**Public Repository**: The repository is intended to be shared across teams and potentially open-sourced. No sensitive information should ever be committed.

**Access Control**: TBD - depends on hosting platform (GitHub, GitLab, Bitbucket, etc.)

### Security Rules Enforced

**BR-023**: No hardcoded secrets, URLs, or environment-specific values are allowed in code.

**BR-024**: Secrets must be externalized to environment variables or a vault.

**BR-025**: Logging must not include passwords, tokens, or personally identifiable information (PII).

These rules are enforced by AI assistants when reviewing or generating code, but the repository itself contains no secrets to protect.
